## 项目说明

当前项目为前端项目，基于 Ant Design Pro 框架开发，用于业务后台 / 中台系统。

后续所有 Agent 在修改本项目代码前，必须先阅读并遵守本文件中的规范。

---

## 一、Agent 基本工作原则

### 1. 修改前必须先理解现有结构

在修改代码前，必须优先阅读以下内容：
- `docs/components` 中对应的组件使用说明
- `docs/issue` 中相关问题记录
- 根目录 `version.md`， 每次写入前都必须查看当前开发版本

禁止在未理解现有实现的情况下直接重构。

### 2. 优先复用现有能力

开发新功能时，应优先复用：

- Ant Design Pro 现有布局能力
- Ant Design 组件
- ProComponents 组件
- 项目已有全局组件
- 项目已有 hooks
- 项目已有 utils
- 项目已有 request / service 封装
- 项目已有权限、菜单、路由、字典、枚举逻辑

禁止重复造轮子。

### 3. 控制改动范围

每次修改应聚焦当前需求，避免无关改动。

禁止：

- 无需求重构大范围代码
- 无理由调整目录结构
- 无理由更换 UI 组件库
- 无理由引入大型依赖
- 无理由修改全局样式
- 无理由修改构建配置
- 无理由修改权限、路由、菜单、登录逻辑

## 二、项目目录约定

当前项目增加 `docs` 目录，用于维护前端开发相关说明。

推荐目录结构如下：

```text
docs/
├── api/
│   ├── user_api.md
│   └── ...
├── issue/
│   └── ...
└── components/
    └── ...
```

### 1. docs/api

用于放置后端提供的接口描述信息，供前端对接使用。

Agent 在新增或修改接口调用前，必须优先查看对应接口文档。

如果接口文档不存在，但本次开发依赖后端接口，Agent 应在回复中明确说明缺少接口文档，不得凭空假设关键字段含义。

### 2. docs/issue

用于记录开发过程中出现的问题、坑点、兼容性问题、过时方法等。

文件命名规则：

全局通用问题统一记录在 docs/issue/global.md
功能模块问题按功能区分文件名
例如：
docs/issue/menu.md
docs/issue/table.md
docs/issue/form.md
docs/issue/auth.md
docs/issue/dashboard.md

Agent 在遇到以下情况时，应补充 issue 记录：

修复了一个容易重复出现的问题
发现某个旧方法已过时
发现某种写法在当前项目中不可用
发现 Ant Design Pro / ProComponents 使用限制
发现接口字段、权限、路由、菜单相关坑点
记录格式建议：
```md
## 问题标题

### 出现场景

说明在哪个功能、哪个页面、什么操作下出现。

### 问题原因

说明导致问题的原因。

### 解决方式

说明最终采用的解决方案。

### 影响范围

说明影响的页面、组件或模块。

### 相关文件

- `src/pages/xxx/index.tsx`
- `src/components/xxx/index.tsx`
```

### 3. docs/components

用于记录项目自定义全局组件的使用介绍。

Agent 在使用全局组件前，必须先查看该目录下对应组件说明。

组件文档应包含：

组件名称
组件用途
使用场景
Props 说明
示例代码
注意事项
已知限制

如果新增了可复用的全局组件，应同步在 docs/components 中新增说明文档。

## 版本更新日志

在docs/version文件下，写入当前的变更内容到最新的版本名称文件中，文件格式为 vx.x.x

### 1. 记录粒度

- 一次功能改动对应一条日志。
- 日志必须具体说明：做了什么 + 影响范围。
- 避免空泛描述，例如：
  - 不推荐：`优化了体验`
  - 推荐：`优化版本检查频率，增加 60 秒节流，减少重复请求`
- 如涉及关键文件，建议在末尾补充文件路径，例如：
  - `- 新增版本一致性检查，菜单切换时发现新版本自动整页刷新（src/app.tsx）`

### 2. 分类标准

每个版本下按以下分类记录：

#### 新增功能

用于记录全新能力、页面、接口、交互、模块。

#### 优化功能

用于记录已有功能的性能、体验、结构、可维护性改进。

#### Bug 修复

用于记录明确的问题修复，包括异常、边界问题、错误逻辑、兼容性问题。
