# 基于 Ant Design Pro 的前端问题排查流程（简要）

本文档用于快速统一前端排查路径，适用于 `React + Ant Design Pro + ProTable` 场景。

## 1. 先定义“问题边界”

- 明确是 **UI 表现异常**、**状态管理异常** 还是 **接口数据异常**。
- 先复现最小路径：页面、操作步骤、期望结果、实际结果。
- 固定变量（账号、时间范围、筛选条件、浏览器）以保证可重复。

## 2. 组件层优先（先看前端状态）

- 检查受控/非受控是否混用（如 `value/defaultValue`、`sortOrder`）。
- 检查关键字段是否稳定：`key`、`dataIndex`、`rowKey`。
- 检查 `useMemo/useEffect` 依赖是否完整，避免“状态变了 UI 不刷新”。
- 对 ProTable 场景，重点看 `onChange`、`pagination`、`sorter` 的联动。

## 3. 数据流排查（再看请求）

- 在发送请求前打印或断点确认 query 参数是否正确。
- 检查字段映射：前端 camelCase -> 后端 snake_case。
- 检查枚举映射：如 `ascend/descend` -> `asc/desc`。
- 确认分页、排序、筛选是否同时生效且互不覆盖。

## 4. 接口层校验（最后看后端）

- 使用 Network 对比请求参数与响应数据结构。
- 确认接口是否支持当前能力（例如是否支持服务端排序）。
- 对同一请求参数做重复验证，排除缓存或幂等差异。

## 5. 常见高频误区

- 请求参数正确，但 UI 状态未受控（图标/选中态不变）。
- 动态列 `key` 与排序字段来源不一致，导致排序状态错位。
- 本地缓存（localStorage）脏数据污染当前页面状态。
- 修了业务映射，但漏了组件层状态回填。

## 6. 最小修复与回归建议

- 修复时尽量做“最小改动”，避免同时改动多个不相关逻辑。
- 至少回归三类场景：初次进入、切换筛选后、刷新后恢复。
- 对表格类问题固定回归：分页、排序、筛选、重置、视图保存/恢复。
- 提交前执行类型检查：`npm run tsc`。

## 7. 推荐排查顺序（一句话版）

- 先组件状态（受控与 key），再参数映射（请求前），最后接口响应（Network）。
